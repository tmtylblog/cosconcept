01 — The Architecture

Two databases, one truth

Collective OS uses a dual-database architecture. PostgreSQL (Neon) is the system of record for all transactional data — auth, billing, profiles, CRUD operations. Neo4j (Aura) is the relationship engine that maps how firms, experts, skills, and industries connect. Some data lives in both, synced via background jobs.

How the sync works

When a firm is enriched (website scraped, PDL queried, AI classified), an Inngest background job writes the results to Neo4j via graph-writer.ts. PostgreSQL remains the source of truth — Neo4j gets a projected copy optimized for graph queries. The graph_node_id column in PG tables enables cross-referencing.

05 — Search Architecture

Three-layer cascading search

The search engine uses a cost-optimized cascade. Neo4j does the heavy lifting for free, vector search narrows the field cheaply, and the LLM only ranks the final shortlist. 

  User Query: "Find a B2B SaaS growth agency in APAC with Salesforce expertise"

  │
  ▼

  LAYER 1 — Neo4j Structured Filter                   ~500 candidates  |  $0.00
  Cypher query traverses HAS_SKILL, IN_CATEGORY,
  SERVES_INDUSTRY, OPERATES_IN edges

  │
  ▼

  LAYER 2 — pgvector Similarity                        ~50 candidates   |  ~$0.001
  Cosine similarity on abstraction profile
  embeddings (1536-dim OpenAI vectors)
  * NOT YET LIVE — falls back to text overlap *

  │
  ▼

  LAYER 3 — LLM Deep Ranking (Gemini)                  ~15 results      |  ~$0.01-0.05
  Bidirectional fit scoring, "why this match"
  explanations, partnership readiness check

  │
  ▼

  Results: Ranked firms with match scores, explanations, and evidence

  Total cost per search: < $0.10  |  Target latency: < 3 seconds

Missing Capabilities

Critical pgvector not live

The embedding column on abstraction_profiles is commented out. Layer 2 search falls back to naive text overlap instead of cosine similarity.

Critical No global firm database

ServiceFirm nodes only exist for enriched platform members. The design envisions 1.5M+ firms. Without this, the graph is sparse.

High Graph sync is shallow

The graph-sync-firm Inngest function only passes firmId, name, website — not the full enrichment payload. Enrichment data doesn't flow to Neo4j properly.

High Abstraction profiles incomplete

The enrich/firm-abstraction event type is defined but has no handler function. Auto-generation on enrichment is not wired.

Medium Trust path not implemented

The 3-hop trust computation (user → knows someone → knows someone → at firm) is design-only. Requires social graph edges first.

Medium Bidirectional matching incomplete

Type definitions exist (bidirectionalFit on MatchCandidate) but the scoring is not computed. Both parties' preferences must align.

07 — Before Bulk Ingestion

Decisions to make

Before ingesting large volumes of data, these architectural questions need answers. The current system handles individual firm enrichment well, but bulk ingestion requires different patterns.

Key questions for the team
  1. Where does bulk data land first? — PostgreSQL import tables (staged, then synced to Neo4j) or directly into Neo4j? The current pattern is PG-first via the imported_* tables.
  2. What's the batch sync strategy? — The current graph-writer.ts handles one firm at a time via Inngest. Bulk ingestion of 10K+ firms needs a batch pipeline (concurrent Inngest jobs? Direct Cypher batch inserts?).
  3. Which edges matter most for launch? — Skills, industries, and markets are the foundation for Layer 1 search. Without those edges, firms are invisible to the matching engine.
  4. Do we need Project nodes? — If ingesting real commercial data (contracts, deliverables, revenue), the Project node type should be built first. This is the "commercial truth" signal.
  5. How do we handle deduplication? — Client nodes use name-based uniqueness. Firms match on website domain. What happens when ingested data has slightly different names for the same entity?
  6. What's the enrichment budget? — Each firm enrichment costs ~$0.01-0.05 (AI classification + PDL). At 10K firms, that's $100-500. At 100K firms, $1,000-5,000.

Current Data Flow (per firm)

  1. User submits domain
     │
     ├── PDL lookup (company data, employee count, industry)
     ├── Jina scrape (website content, services, team, clients)
     │
     ▼
  2. AI Classification (Gemini Flash)
     │
     ├── Map to COS skills (L2 level)
     ├── Map to industries
     ├── Map to categories (firm archetype)
     ├── Extract clients, services, team members
     │
     ▼
  3. Persist to PostgreSQL
     │
     ├── service_firms.enrichment_data (full JSON)
     ├── enrichment_audit_log (trail)
     │
     ▼
  4. Sync to Neo4j (via Inngest background job)
     │
     ├── Create ServiceFirm node
     ├── Create edges: IN_CATEGORY, HAS_SKILL, SERVES_INDUSTRY, etc.
     ├── Create Client nodes + HAS_CLIENT edges
     ├── Create Service nodes + OFFERS_SERVICE edges
     │
     ▼
  5. Deep enrichment (background, progressive)
     │
     ├── Deep crawl (up to 30 pages for case studies, team bios)
     ├── Case study ingestion (AI analysis of each case study URL)
     ├── Expert LinkedIn enrichment (PDL person lookup per expert)
     └── Abstraction profile (hidden AI-generated profile for matching)
Recommendation for bulk ingestion

The safest path is: import to PostgreSQL imported_companies table first → classify and enrich in batches via Inngest → sync to Neo4j. This preserves the audit trail and allows re-processing if the classification model improves. The existing import scripts (scripts/import-companies.mjs) follow this pattern.
